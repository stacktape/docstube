## How it works

1. Point docstube at your repo. A local web UI wizard opens.
2. Choose the doc type, define reader personas, add a little context.
3. A writer agent reads your code; reviewer and verifier agents check it for audience fit and accuracy.
4. Review the result rendered exactly as it'll look live, and leave feedback on any element.
5. The writer agent refines until you're happy.
6. On every code change, only the affected pages regenerate.

## The gate — how it proves it

This is the "prove it" half, and it's not a vibe check. Before a page ships, it has to pass:

- If a code sample doesn't compile, the page doesn't ship. (`tsc` / `pyright`.)
- Imports resolve against the real package — phantom functions never reach the reader.
- Broken links, invalid MDX, and malformed D2 diagrams are machine-checked. They don't reach readers.
- API tables are pulled from your types, not written by an LLM. Signatures change, tables update — zero tokens.
- The agent that writes the page isn't the agent that judges it.
- No page ships with an open blocker. That's not a policy — it's the build.

docstube can also run a drift report: it'll tell you exactly where your existing docs disagree with your code.

## FAQ

**"Write themselves" sounds like a recipe for slop.** That's why the second half exists. Nothing ships until it clears deterministic checks and cross-vendor review — and you review the rendered result before it goes live.

**Who actually checks the writer?** A separate verifier whose model family differs from the writer's, plus deterministic checks that don't care what any model thinks.

**Does my code leave my machine?** Source context goes only to the AI provider you already use, on your own credentials — never to docstube's servers.

**What about code samples?** Every TS sample runs through `tsc` and every Python sample through `pyright`. If it doesn't compile, the page doesn't ship.
