## Why nobody trusts the wiki

Every team has the doc that's probably wrong. So new hires don't rely on it — they ping the one engineer who knows, who answers the same question for the fifth time this month. The docs exist; the trust doesn't. And untrusted docs don't get read, which means they don't get fixed, which means they get less trusted. That's the loop docstube breaks.

It breaks it by changing what "trust" rests on. You don't trust a teammate's memory of when they last updated the page. You trust that the page cleared a verification gate built from the actual code.

## Verified is what makes it trusted

Internal readers stop second-guessing docs that can't contain a claim the code contradicts. Before any internal page lands, deterministic checks run, and a model from a different family reviews what the writer wrote.

- **If a code sample doesn't compile, the page doesn't ship.** `tsc` for TS/JS, `pyright` for Python — so the snippet a new hire pastes actually runs.
- **Imports resolve against the real package — phantom functions never reach the reader.** No "this method used to exist."
- **API tables are pulled from your types, not written by an LLM,** so the internal reference matches the internal code.
- **Broken links and malformed diagrams are machine-checked,** so the architecture diagram isn't quietly out of date.
- **No page ships with an open blocker — not a policy, the build.**

A doc that can't be confidently wrong is a doc your team will actually open.

## Capture the knowledge that lives in three people's heads

The most valuable internal docs are the ones no one has time to write: how the auth flow really works, why the job queue is shaped that way, what the deploy actually does. docstube plans coverage from a tree-sitter codemap, so the page list comes from what's in the codebase — not from what someone remembered to document. Architecture overviews and onboarding paths get drafted from real source, with D2 diagrams compiled as part of the gate. What it can't ground in code, it surfaces for review instead of inventing.

## It doesn't go stale the week after onboarding

Internal docs rot fastest, because the code under them moves daily. docstube keeps symbol-level provenance via normalized hashes, so when a function changes, only the pages that depend on it regenerate. Wire up the GitHub Action and it opens a PR with just the changed pages and the reasons — reviewed like any other change, never pushed silently. A drift report tells you where the wiki already disagrees with `main`, so you find the rot before the next hire does.

## Private repos, your infra, your keys

This is internal, so the privacy model matters. docstube runs locally and on your own CI, against your private repos. Its own servers never receive your source, your prompts, your generated docs, your file paths, or your project names. Your source context goes only to the AI provider you already use, on your own credentials — never to docstube's servers. If that provider is a self-hosted or open-weight model behind your own endpoint, that's just a config setting.

## How it works

1. Point docstube at your repo. A local web UI wizard opens.
2. Choose the doc type, define your reader personas, add a little context.
3. A writer agent reads your code; reviewer and verifier agents check it for fit and accuracy.
4. Review the result rendered exactly as it'll look live, and leave feedback on any element.
5. The writer agent refines until you're happy.
6. On every code change, only the affected pages regenerate.

The wizard asks the minimum, then runs on its own. Your leads steer the depth; nobody has to babysit a doc.

## FAQ

**Why would my team trust generated docs more than the wiki we have?** Because these clear a verification gate the wiki never had. Code samples compile, imports resolve, and API tables come straight from your types — a page can't carry a claim the code contradicts.

**Does our private source go to docstube?** No. docstube's servers never receive your source, prompts, generated docs, paths, or project names. Source context goes only to the AI provider you already use, on your own credentials.

**Can we run it fully on our own models?** Yes. Point the direct-API adapter at a self-hosted or open-weight endpoint with your own base URL and model. It's a config setting, not a special integration.

**How do we keep it from going stale?** Run the GitHub Action. When code changes, it regenerates only the affected pages and opens a PR with the reasons, so updates get reviewed like any other change.
